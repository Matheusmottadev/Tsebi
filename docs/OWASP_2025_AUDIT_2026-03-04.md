# Relatório OWASP 2025 — Tsebi
Data: 2026-03-04
Escopo: Backend `server/` (Node/Express), frontend `web-next/` (Next.js + legacy), rotas/API, webhooks e integrações.

## Sumário Executivo (Top 10 achados por risco)
1. **[Critical][A01] IDOR em consulta de pedido por `orderId` + `email` via query** (`GET /api/orders/:orderId`).
2. **[High][A01] Exposição de recomendações por usuário via `userId` controlado pelo cliente** (`GET /api/recommendations`).
3. **[High][A08] Webhook WhatsApp sem verificação de assinatura no POST**.
4. **[High][A07/A02] Token admin VIP aceito em query string (`?token=`)**.
5. **[High][A08] Endpoint `/api/meta/capi` acessível ao cliente usando token CAPI do servidor (abuso de integridade/telemetria).**
6. **[Medium][A10/A08] Webhook Stripe responde 200 até para assinatura ausente/inválida (monitoramento e fail-open de validação).**
7. **[Medium][A04] PII sensível (CPF/telefone) armazenada em texto claro no banco (sem evidência de criptografia de coluna).**
8. **[Medium][A02] CSP com `unsafe-inline` em `script-src` reduz proteção contra XSS.**
9. **[Medium][A09] Falta de `requestId/correlationId` padronizado em logs de segurança.**
10. **[Low][A03] GitHub Actions sem pin por SHA (supply-chain hardening incompleto).**

## A01 — Broken Access Control
### Checklist
- [x] Autorização server-side por recurso (owner check)
- [x] Verificar IDOR em endpoints por `id`
- [x] Verificar uso de IDs sensíveis vindos do cliente

### Evidências
- **IDOR de pedido por email**
  - `server/index.ts:2426-2435`: autorização alternativa por `email` query (`hasEmailAccess`) sem sessão.
- **Recomendação de outro usuário**
  - `server/index.ts:1917-1920`: `requestedUserId` vindo de query tem precedência sobre sessão.
- **Controle positivo observado**
  - `src/routes/shipping.routes.ts:55-58`: owner check correto em quotes por pedido.

### Exploit plausível
1. Atacante obtém `orderId` (URL, print, e-mail vazado) e e-mail da vítima; consulta status/dados do pedido sem login.
2. Atacante chama `/api/recommendations?userId=<alvo>` e extrai preferências/comportamento da conta alvo.

### Correção recomendada
- Em `/api/orders/:orderId`, remover fallback por e-mail puro. Usar:
  - sessão autenticada **ou**
  - token de acesso de pedido assinado e com expiração curta.
- Em `/api/recommendations`, ignorar `req.query.userId` para usuários comuns; usar apenas `req.session.userId`.

### Patch sugerido (diff)
```diff
--- a/server/index.ts
+++ b/server/index.ts
@@
- const requestedUserId = String(req.query.userId || "").trim();
- const userId = requestedUserId || sessionUserId;
+ const requestedUserId = String(req.query.userId || "").trim();
+ const userId = sessionUserId;
+ if (requestedUserId && requestedUserId !== sessionUserId) {
+   return res.status(403).json({ error: "FORBIDDEN_USER_SCOPE" });
+ }
@@
- const requestedEmail = normalizeEmail(req.query?.email || "");
- const orderEmail = normalizeEmail(order.userEmail || "");
- const hasEmailAccess = Boolean(requestedEmail && orderEmail && requestedEmail === orderEmail);
- if (!hasSessionAccess && !hasEmailAccess) {
+ if (!hasSessionAccess) {
    return res.status(401).json({ error: "UNAUTHORIZED" });
 }
```

### Testes a criar/rodar
- Integração: usuário A não consegue acessar pedido de B por `orderId+email`.
- Integração: `/api/recommendations?userId=B` retorna 403 para usuário A.

## A02 — Security Misconfiguration
### Checklist
- [x] CORS
- [x] Helmet/CSP/HSTS
- [x] cookies (`HttpOnly`,`Secure`,`SameSite`)
- [x] limites de payload/upload/rate-limit

### Evidências
- CORS por allowlist: `server/index.ts:1351-1358`.
- Helmet + CSP ativo: `server/index.ts:1362+`.
- Sessão segura em produção: `server/session.ts:64-67`.
- CSRF token cookie não `HttpOnly` (necessário para double-submit): `server/middlewares/userCsrf.ts:152-160`.
- CSP permissiva em scripts (`'unsafe-inline'`): `server/index.ts:1371-1374`.

### Exploit plausível
1. XSS em qualquer ponto do front ganha maior chance de execução com inline script permitido.
2. Config fraca de CSP aumenta superfície para injeções DOM.

### Correção recomendada
- Migrar CSP para nonce/hash em inline scripts e remover `'unsafe-inline'` de `script-src`.
- Revisar `frame-ancestors` conforme necessidade de embeds externos.

### Patch sugerido (diff)
```diff
--- a/server/index.ts
+++ b/server/index.ts
@@
- scriptSrc: ["'self'", "'unsafe-inline'", ...]
+ scriptSrc: ["'self'", ...]
+ // usar nonce por request para scripts inline necessários
```

### Testes
- E2E: páginas carregam sem violação CSP após nonce/hash.
- Security test: injeção inline simples bloqueada pelo browser CSP.

## A03 — Software Supply Chain Failures
### Checklist
- [x] lockfiles e `npm audit`
- [x] CI/CD hardening
- [x] secrets no repo

### Evidências
- `npm audit` (root + web-next): **0 vulnerabilidades** reportadas.
- CI usa actions por tag major (`@v4`) sem pin SHA: `.github/workflows/ci.yml`.
- `.env` e `web-next/.env.local` contêm chaves reais **no workspace local**; não estão trackeados por `git ls-files` (confirmado).

### Exploit plausível
1. Compromisso upstream em action não pinada por SHA.
2. Vazamento operacional de segredos locais por backup/sync indevido fora do Git.

### Correção recomendada
- Pin de actions por SHA digest.
- Secret scanning automatizado (gitleaks/trufflehog) no CI.
- Rotacionar chaves reais que já foram expostas localmente em ambiente compartilhado.

### Patch sugerido
```diff
- uses: actions/checkout@v4
+ uses: actions/checkout@<sha-pinned>
```

### Testes
- CI job de secret scan.
- CI valida lockfile e bloqueia `npm install` fora de `npm ci`.

## A04 — Cryptographic Failures
### Checklist
- [x] hashing de senha
- [x] segredo de sessão
- [x] PII em repouso/transporte

### Evidências
- Senhas com bcrypt(12): `server/auth.ts:915`, `1247`, `1354`.
- Enforça `SESSION_SECRET` forte em produção: `server/session.ts`.
- PII (CPF/telefone/email) persistida em várias tabelas/objetos sem evidência de criptografia de coluna (ex.: `server/auth.ts:118-119`, `1481`, `server/lib/vip-repository.ts`).

### Exploit plausível
1. Vazamento de banco expõe CPF/telefone em claro.

### Correção recomendada
- Criptografia de campos sensíveis em repouso (ex. pgcrypto ou envelope encryption KMS).
- Minimização e retenção: armazenar apenas o necessário.

### Patch sugerido
- Incremental: mascarar CPF em respostas por padrão; liberar completo só quando estritamente necessário e autorizado.

### Testes
- Unit: serialização de resposta nunca inclui CPF completo fora de contexto autorizado.

## A06 — Insecure Design
### Checklist
- [x] validações de fluxo de checkout
- [x] idempotência e transições de estado
- [x] abuso de endpoint de telemetria

### Evidências
- Idempotência webhook Stripe via `webhook_events`: `server/index.ts:888-899`, `1021-1027`.
- Endpoint `/api/events` aceita `userId` do cliente quando sessão ausente: `server/index.ts:1634-1636` (risco de envenenamento analítico).
- `/api/meta/capi` pode ser acionado por cliente com event data: `server/index.ts:1656-1683`.

### Exploit plausível
1. Envenenamento de perfil/recomendação via eventos falsos.
2. Fraude de mensuração em Meta CAPI (evento server-side indevido).

### Correção recomendada
- `userId` de eventos sempre vindo da sessão (ignorar body).
- Restringir `/api/meta/capi` a chamadas internas autenticadas (`x-internal-api-key` ou remover endpoint público).

### Patch sugerido
```diff
--- a/server/index.ts
+++ b/server/index.ts
@@
- userId: payload.userId || String(req.session?.userId || "").trim(),
+ userId: String(req.session?.userId || "").trim(),
@@
-app.post("/api/meta/capi", ...
+app.post("/api/meta/capi", requireInternalApiKey, ...
```

### Testes
- Integração: request sem sessão não consegue setar `userId` em `/api/events`.
- Integração: `/api/meta/capi` sem chave interna retorna 401/403.

## A08 — Software or Data Integrity Failures
### Checklist
- [x] assinatura de webhooks
- [x] deduplicação/idempotência
- [x] validação de payload confiado

### Evidências
- Stripe webhook assinado com `constructEvent`: `server/index.ts:1474-1479`.
- Deduplicação de evento Stripe: `server/index.ts:891-894`.
- **Falha**: WhatsApp webhook POST não verifica assinatura (`X-Hub-Signature-256`): `src/routes/whatsapp.routes.ts:35-72`.
- **Falha de postura**: webhook Stripe retorna 200 para assinatura ausente/inválida: `server/index.ts:1464-1479`.

### Exploit plausível
1. Envio de payload WhatsApp falso contaminando base e automações.
2. Flood de requests inválidos no webhook Stripe sem visibilidade de falha real no status HTTP.

### Correção recomendada
- Verificar assinatura HMAC SHA256 do WhatsApp Cloud API.
- Retornar 400 para assinatura Stripe ausente/inválida (mantendo 200 apenas para eventos já processados/ignorados válidos).

### Patch sugerido
```diff
--- a/src/routes/whatsapp.routes.ts
+++ b/src/routes/whatsapp.routes.ts
@@
+ const crypto = require("crypto");
+ function verifyWhatsAppSignature(req) {
+   const sig = String(req.headers["x-hub-signature-256"] || "");
+   const appSecret = String(process.env.WHATSAPP_APP_SECRET || "").trim();
+   if (!sig || !appSecret) return false;
+   const expected = "sha256=" + crypto.createHmac("sha256", appSecret).update(JSON.stringify(req.body || {})).digest("hex");
+   return crypto.timingSafeEqual(Buffer.from(sig), Buffer.from(expected));
+ }
@@
-whatsappRouter.post("/webhook", async (req, res) => {
+whatsappRouter.post("/webhook", async (req, res) => {
+  if (!verifyWhatsAppSignature(req)) return res.status(401).json({ ok:false, error:"INVALID_SIGNATURE" });
```

### Testes
- Integração: WhatsApp webhook sem assinatura -> 401.
- Integração: assinatura correta -> 200.

## A09 — Security Logging and Alerting Failures
### Checklist
- [x] logs de auth/webhook/pagamento
- [x] correlação entre logs
- [x] vazamento de PII/segredos

### Evidências
- Há logs de ciclo Stripe (`logStripeLifecycle`) e Meta (`meta_capi_request/response`).
- Não há padrão explícito de `requestId` propagado em todas as requisições.
- Logs podem incluir stack completa em alguns pontos (`server/index.ts:1344`).

### Exploit plausível
1. Incidente de fraude sem trilha correlacionável por request.
2. Excesso de detalhe em stack no log pode vazar informação operacional.

### Correção recomendada
- Middleware de `requestId` (`x-request-id`) + inclusão em todos logs críticos.
- Redaction padrão para email/CPF/token.

### Patch sugerido
```diff
+ app.use((req,res,next)=>{
+   req.requestId = String(req.headers['x-request-id'] || crypto.randomUUID());
+   res.setHeader('x-request-id', req.requestId);
+   next();
+ });
```

### Testes
- Integração: toda resposta inclui `x-request-id`.
- Unit: logger inclui requestId em eventos críticos.

## A10 — Mishandling of Exceptional Conditions
### Checklist
- [x] fail-open/fail-closed em integrações
- [x] consistência de status de pedido
- [x] retry/idempotência

### Evidências
- Positivo: bloco de analytics não bloqueia webhook de pagamento (comentado e implementado): `server/index.ts:1285+`.
- Risco: webhook Stripe retorna 200 em assinatura inválida/ausente (`server/index.ts:1468-1479`), mascarando erro operacional.
- Teste API falhando por expectativa de 400 vs retorno 200 (`tests/api/backend.api.test.ts:979`).

### Exploit plausível
1. Falha de configuração de assinatura passa despercebida em monitoria HTTP.

### Correção recomendada
- Ajustar resposta para 400 em assinatura inválida/ausente.
- Atualizar testes e alertas para diferenciar `invalid_signature` de eventos processados.

### Patch sugerido
```diff
- return res.status(200).json({ received: true, ignored: "missing_signature_header" });
+ return res.status(400).json({ received: false, error: "missing_signature_header" });
@@
- return res.status(200).json({ received: true, ignored: "invalid_signature" });
+ return res.status(400).json({ received: false, error: "invalid_signature" });
```

### Testes
- Atualizar `tests/api/backend.api.test.ts` para assinatura inválida => 400.
- Cobrir caso assinatura válida + evento duplicado => 200 sem reprocessar.
